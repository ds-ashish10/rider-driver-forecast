Case study steps:

  1. Data processing and Cleansing (Null value treatment, de-duplication etc.)

  2. Understanding of Data and EDA (Exploratory data analysis)

  3. Data pre-processing:

    i) Transformation of Datetime stamp to Day, Weekday, Month and Year for analysis and forecast

    ii) Geofencing : Creation of a virtual boundary and classification of the entire operational area (using ride start loc.) into 10 grid boxes (Loc_B) for the demand forecast and 100 grid boxes (Loc_S) for distribution of the demand forecast on the basis of their ride_value contribution over the previous week. (see detailed explanation in "Feature Engineering" section).

    Note: square grid boxes have been used for this case study but polygon geofencing would be better because it covers the high density area and discards the non-active area on the map.

    Note: The grid boxes (Loc_B) that have less than 1% ride_value have been removed from the analysis as they cover the low demand area.

  4. Demand Forecasting:

    i) Generate Demand Forecast across 10 grid big boxes(Loc_B) [Remove the grid boxes has less than 1% ride_value]. ARIMA and ETS have been used for generation of demand forecast.

    ii) Distribution of demand forecast over 100 small grid box (Loc_S) on the basis of their last week contribution.

    iii) Accuracy calculation of ARIMA and ETS iv) Generation of demand forecast using XGBoost Regressor for better result and comparison of accuracy with ARIMA and ETS.

    Note: Daily level data has been used to generate demand forecast (as the data frequency is not mentioned in the task). In case of actual deployment at hourly level, the daily level data can be distributed on an hourly basis based on every weekday of the week.

# Creation of grid cell based on start location. (Loc_S)

## Create GeoDataFrame for start locations
df["start_geometry"] = df.apply(lambda row: Point(row["start_lng"], row["start_lat"]), axis=1)
gdf_start = gpd.GeoDataFrame(df, geometry="start_geometry")
gdf_start.set_crs(epsg=4326, inplace=True)

## Create a bounding box for the start locations
start_points = gdf_start.geometry.unary_union
min_lon, min_lat, max_lon, max_lat = start_points.bounds

## Define number of grid cells (1000 boxes)
num_boxes = 100

## Compute grid size based on bounding box dimensions
width_km = geodesic((min_lat, min_lon), (min_lat, max_lon)).km
height_km = geodesic((min_lat, min_lon), (max_lat, min_lon)).km
grid_width = width_km / np.sqrt(num_boxes)
grid_height = height_km / np.sqrt(num_boxes)

## Compute lat/lon step sizes
lat_step = (max_lat - min_lat) / np.sqrt(num_boxes)
lon_step = (max_lon - min_lon) / np.sqrt(num_boxes)

## Function to assign a grid cell ID to each start location
def assign_grid_cell(lat, lon):
    lat_index = int((lat - min_lat) / lat_step)
    lon_index = int((lon - min_lon) / lon_step)
    return lat_index * int(np.sqrt(num_boxes)) + lon_index

## Apply the function to assign grid cell IDs
df["grid_box_id"] = df.apply(lambda row: assign_grid_cell(row["start_lat"], row["start_lng"]), axis=1)

def forecast_last_week(df):
    forecasts = []
    unique_grid_boxes = df['grid_box_id2'].unique()

    for grid_id in unique_grid_boxes:
        df_grid = df[df['grid_box_id2'] == grid_id].set_index('date').resample('D').sum()

        # Ensure sufficient data for forecasting
        if len(df_grid) < 14:
            continue

        # Exponential Smoothing
        model_es = ExponentialSmoothing(df_grid['ride_value'], seasonal='add', seasonal_periods=7)
        fit_es = model_es.fit()
        forecast_es = fit_es.forecast(7)  # Predict last 7 days

        # ARIMA Model
        model_arima = ARIMA(df_grid['ride_value'], order=(2,1,2))
        fit_arima = model_arima.fit()
        forecast_arima = fit_arima.forecast(7)  # Predict last 7 days

        # Store forecast results
        forecast_df = pd.DataFrame({
            'date': df_grid.index[-7:],  # Last week dates
            'grid_box_id2': grid_id,
            'forecast_es': forecast_es.values,
            'forecast_arima': forecast_arima.values,
        })
        forecasts.append(forecast_df)

    return pd.concat(forecasts, ignore_index=True)

# Feature Engineering: Creating Lag Features & Rolling Statistics for XGBoost
def create_features(data):
    """Creates lag and rolling statistics features for time series forecasting."""
    for lag in [1, 7, 14]:
        data[f'lag_{lag}'] = data.groupby('grid_box_id2')['ride_value'].shift(lag)
    
    # Rolling Mean & Standard Deviation
    data['rolling_mean_7'] = data.groupby('grid_box_id2')['ride_value'].transform(lambda x: x.rolling(7).mean())
    data['rolling_std_7'] = data.groupby('grid_box_id2')['ride_value'].transform(lambda x: x.rolling(7).std())
    
    return data

def forecast_last_week(df):
    forecasts = []
    unique_grid_boxes = df['grid_box_id2'].unique()

    for grid_id in unique_grid_boxes:
        df_grid = df[df['grid_box_id2'] == grid_id].set_index('date').resample('D').sum()

        # Ensure sufficient data for forecasting
        if len(df_grid) < 14:
            continue

        # Exponential Smoothing
        model_es = ExponentialSmoothing(df_grid['ride_value'], seasonal='add', seasonal_periods=7)
        fit_es = model_es.fit()
        forecast_es = fit_es.forecast(7)

        # ARIMA Model
        model_arima = ARIMA(df_grid['ride_value'], order=(2,1,2))
        fit_arima = model_arima.fit()
        forecast_arima = fit_arima.forecast(7)

        # Store forecast results
        forecast_df = pd.DataFrame({
            'date': df_grid.index[-7:],  
            'grid_box_id2': grid_id,
            'forecast_es': forecast_es.values,
            'forecast_arima': forecast_arima.values,
        })
        forecasts.append(forecast_df)

    return pd.concat(forecasts, ignore_index=True)

Conclusion of the Case study:

  1. Results:
    i) On the basis of accuracy results: The Exponential Smoothing (ETS) Algorithm has the better results over ARIMA and XGBoost.
    Simple algorithms perfoms better in case of less data.
    ii) XGBoost does't perform well due to less historical data (only 3 weeks data are available) and features, it may perform better if more historical data available (atleast 6 months to a year).
    iii) The accuracy has been calculated on daily basis and it is a direct calculation instead of weighted (on the basis of contribution). The weighted accuracy can look a little better at hourly basis.

  2. Deployement:
    i) For the deployement, we need to run the model daily with data of previous day extended until historically avaiable.
    ii) The final demand forecast needs to be distributed hourly for every location (Loc_S) so that the driver knows where and when they need to move for more ride.
    iii) In the case study 'ride_value' has been taken instead of "ride_count" asuming that high value with low count is better than low value with high count. It means the case study is directly focusing on the monetary aspect. If the count of rides accounts to more profit to company then ride_value should be replaced by count of rides.

3. Scope and Challenges:

This is simple case study with limited data, but the actual real-life scenario would be different. Using more complex algorithms and data we can improve the profitability with optimization, following are some recommendations:

1. More historical data needed. For eg. ride end time is also an important feature.
2. The availability of weather data significantly impact the demand of riders. 
3. Most taxi companies uses surge pricing, surge pricing data trends can be analyzed to anticipate high-demand areas.
4. Important events (like sports), concert and local festivals information can be useful for analysing high demand area and time
5. Since the data is very flat, if we have Customer and Driver IDs then we can recommend the rides to the driver more efficiently. 
6. Use of geographical tool like ArcGIS create more impact in solution, as these tools have excelent GIS capabilities. e.g. POINT Geometry and geodataframe with CRS 4326 is better than making a square grid box. 
7. Some more important features like details of the ride (premium/economy, distance travel, riders/driver details with rating, area information like airport etc.) impacts the recomendation.
8. More features leads to multivariate analysis which performs better than time-series. 
9. Route optimisation (like TSP problem) is a whole different thing and complex but incorporation of route optimisation and recomendation impcats the demand and value.
